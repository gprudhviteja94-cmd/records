📒Meeting Outline

Recipient Entitlement and UI Behavior

• Button click validation vs. pre-check
  Team discussed whether to validate recipient entitlement on button click or pre-check earlier. Current thinking: leverage existing recipient entitlement already used within the Recipient module rather than reinventing checks.
• Show/hide button logic
  Proposal to derive a flag “isRecipientLookupEnabled” during initial data load. If true, show the “Select Recipient” button; if false, hide it to avoid showing an empty drawer/overlay.
• Empty drawer UX concern
  Showing the button unconditionally could result in an empty drawer for users without entitlement. Consensus leans toward hiding the button when entitlement is absent to avoid poor UX.

Data Flow and Initialization Flags

• Flag sourcing and service calls
  On landing/initialization for Create Payment, backend should call Recipient Management to determine if the user is integrated and set isRecipientLookupEnabled accordingly. The button visibility is driven by this flag.
• Existing APIs and Swagger check
  There is mention of a new API in Recipient Store; team to verify existing coverage and details via Swagger and with the original implementer to avoid duplicating functionality.

API Gateway and Service Topology

• Direct calls within the “house”
  Recipient Lookup from TSP Web to Recipient API is a direct internal call (no API gateway) since services are within the same domain/microservice ecosystem (“inside the house,” port 80894 for Recipient Management).
• User Preferences path
  Updating preferences goes via CS Portal’s APIs; may be behind an API gateway. For TSP it’s acceptable to call their exposed API; gateway presence is an implementation detail on their side.
• Separation of concerns
  Recipient APIs and Preferences APIs are unrelated; patterns can differ given their ownership and exposure model.

Preferences, Re-initialization, and Language Change

• Initialize call usage
  Initialize is used on app load and for refetching UI after preference changes (e.g., language). After changing language in Preferences and navigating back to TSP UI, an initialize call is triggered to refresh context/UI strings.
• Call origin and routing
  Historically, Common Services made two initialize calls (from non-IFA gateway upon login and another flow). In Ember (current implementation), TSP can also issue initialize if user context/preferences aren’t available after navigation, especially when moving between apps/engines where context is lost.
• Risk of over-refresh
  Re-initializing the entire UI can be heavy; ensure initialize is scoped/triggered only when necessary (e.g., post-preference change) to avoid unnecessary data calls and reloads.

Next Steps and Ownership Clarifications

• Consult original implementer
  Connect with Omar (built the recipient class) to confirm entitlement checks, available methods, and whether the “isRecipientLookupEnabled” flag already exists in current APIs.
• Design alignment
  Confirm with design that no recipient entitlement means the recipient button and drawer should not be shown; the review toggle is unrelated and should remain independent.

📋Overview

• Use existing recipient entitlement mechanisms; avoid duplicating checks.
• Drive recipient button visibility via an “isRecipientLookupEnabled” flag resolved during initialization.
• Do not show an empty recipient drawer; hide entry points when entitlement is absent.
• Recipient Lookup: direct internal call from TSP Web to Recipient API (no gateway).
• User Preferences: call CS Portal API; gateway presence is transparent to TSP.
• Initialize is invoked on load and after preference changes to refresh UI; limit unnecessary full refreshes.
• Verify API availability and details via Swagger and with Omar before adding new endpoints.
• Review toggle is unrelated to recipient entitlement and should not be coupled.
