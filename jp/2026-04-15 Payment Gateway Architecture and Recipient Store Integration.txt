📒Meeting Outline

Payment Gateway Architecture and Routing

• End-to-end UI to backend flow
  Participants aligned on the navigation flow: User logs into Common Services (CS) portal, selects a feature (Payments, Templates, Imports, Consolidated Grid, Recipients, Payment Details). Each feature is a micro-frontend (MFE) registered with CS. CS resolves the route-to-app mapping and loads the selected MFE. All calls should traverse the Payment Gateway.

• Gateway behavior for TSP Web vs Recipient Management
  • For TSP Web routes: current gateway passthrough with no filters/security checks (no IDA, no Sentry). Rationale: TSP Web already performs its own checks on VM-hosted services. Once TSP migrates to GKE and adopts common security, gateway filtering will be enabled and this exception removed.
  • For Recipient Management routes: gateway enforces Sentry user validation and ADFS token validation, then forwards to Recipient Management API and returns the response to the caller.

• Scope and future onboarding
  Gateway is designed to front multiple backend services (current: TSP Web, Recipient Management). Other systems can be onboarded easily. All MFEs (Payments, Grid, Import, Templates) must route via the Payment Gateway for uniform architecture.

Recipient Store Integration (Unipax module)

• Purpose and ownership
  Recipient Store is a new external module/service (from Unipax) to provide recipient lists and details for the new UI design. Owned by our team (primarily Sudha) for integration; service operated by Unipax team (Omar).

• Current integration approach
  • Backend (TSP Web) exposes internal endpoints:
    • GET /search-recipients: paginated, filterable, sortable list
    • GET /recipients/{id}: recipient detail by ID
  • Internally, these call a Recipient Management RestClient, which invokes Recipient Store’s external APIs. Retry logic and ID/auth header propagation are implemented.
  • No create/update operations are in scope now; fetch-only.

• Routing placement
  Presently, calls from TSP Web to Recipient Store do not go through our Payment Gateway (treated as internal app-to-app). Recipient Store is currently behind Unipax’s own gateway. Proposal is to standardize via our gateway when they allow routing; otherwise, use service-mesh-style internal gateway in future GKE architecture.

• Data/source clarification
  Existing DSP does not maintain a dedicated beneficiary/recipient master in our workspace; current flows persist payment data (accounts, bank/country/branch tables). Recipient Store remains the source for recipient information.

• Open issues discovered
  • Authentication/authorization: Received 401/403 when calling Recipient Store. Swagger provided lacks auth header schemes; bearer token behavior unclear. Likely needs app-to-app cert/JWT plus user entitlement checks.
  • Entitlements model mismatch: Recipient Store may use Unipax’s “simplified entitlements” model, different from current DSP. Need clarity on required permissions/claims and onboarding path.
  • User onboarding: Our TSP test users are not present/entitled in Recipient Store; empty recipients should return 200 with empty list, not 403.

Common Services (CS) and User Preferences

• Read vs update
  • Current: MFEs read user preferences from CS (e.g., theme, layout, ordering).
  • New product ask: Allow updating CS user preferences from the TSP UI (uplift MFE). CS already has database and APIs (and even UI) for preferences; our role would be to call their update API.

• Concerns and proposed stance
  • Preferences cover multiple apps (Uplift, ITC, GSH, etc.). Updating through one MFE risks scope creep or inadvertently touching others’ data.
  • Principle proposed: CS should own both storage and update UI/API; users update via CS portal. If product insists, constrain our updates strictly to uplift-specific keys, without reading or writing unrelated app preferences.
  • Decision deferred; document both approaches and seek product approval.

Deployment and Platform

• Current state
  • Dev GKE: Payment Gateway deployed.
  • TSP UI (React MFEs): on GKE.
  • TSP Web: remains on VMs; gateway routes directly to provided hostnames.

• Target state
  Unify under one gateway once TSP Web moves to GKE; adopt service mesh for internal service-to-service calls.
