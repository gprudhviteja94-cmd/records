📒Meeting Outline

Architecture & Gateway Strategy

• End-to-end flow across MFEs and Gateway
  The user logs into Common Services (CS) portal and navigates via left navigation to a specific micro frontend (MFE) such as Payments, Templates, Imports, Consolidated Grid, Recipients, or Payment Details. Each app routes via CS-registered routes. The selected MFE is loaded, and all backend calls are proposed to pass through the Payment Gateway. Payment MFE calls TSP Web via the gateway. For TSP Web routes, gateway currently performs pass-through (no IDL/Sentry/ADFS checks); for Recipient Management routes, gateway performs Sentry validation (user), ADFS token validation, then forwards to the Recipient Management API.

• Current exceptions and future state
  Presently, TSP Web is exempted from gateway filtration to avoid duplicating legacy checks already in TSP. Once TSP migrates to GKP and adopts shared controls, the exemption will be removed. Recipient Management currently goes through a different (Unitas) gateway; the team’s stance is to standardize behind one DSP gateway (our Payment Gateway) as systems migrate.

• Internal vs external calls and service mesh
  Calls from UI go through the Payment Gateway. Backend-to-backend internal calls (e.g., TSP Web to Recipient Store) should use internal service mesh (Istio on Kubernetes) rather than the external gateway. External exposures should unify behind a single gateway over time.

Recipient Store Integration

• Scope and ownership
  Recipient Store is a new module (used in Unipax, live for clients) to provide recipient search and details. Owned by Veeam (Uday), primarily worked on by Omar/Omer’s team. Not part of current ESP/TSP databases. Our system will consume it purely as an API/system of record for recipients.

• Why we need Recipient Store
  Current TSP stores beneficiary-like data embedded in payment/template tables; there is no dedicated recipient entity. Recipient Store introduces recipient terminology and centralized management for recipient data needed by new UI (grid, details).

• Routing and exposure
  Recipient Store routing exists in our gateway config but is not yet allowed/enabled by Recipient Store owners. Today, Recipient Management APIs are exposed via their own Unitas gateway. Team position: all consumer access should standardize via our Payment Gateway when feasible; otherwise, internal calls from TSP Web should traverse service mesh.

• Backend integration design (current PR)
  TSP Web adds Recipient Management Controller with two endpoints:
  • Search recipients: paginated, sortable, filterable list; proxies to external Get Recipient Summary API.
  • Get recipient by ID: fetches recipient details; proxies to external Get Recipient Details API.
  Internal client handles request/response mapping, authentication headers (HTTP user), and retry logic aligned with existing patterns. No create/update operations in scope yet; read-only for now.

• Open technical issues discovered
  • Authentication/authorization: External Swagger is missing security scheme declarations. Actual APIs require bearer token; without it returns 401; with token returns 403 for current TSP users. Likely entitlement/tenant/onboarding mismatch.
  • User onboarding: Our dev/test TSP users are not onboarded in Recipient Store; empty result should return empty list (200), not 403. Need confirmation and fixes.
  • Entitlement model mismatch: Recipient Store (from Unipax) uses “simplified entitlements”; our current DSP/TSP uses legacy entitlements. Admin Center must support mapping or dual-mode entitlements; otherwise current users can’t be granted needed permissions. Clarify expected JWT claims and permission strings (e.g., @PreAuthorize hasPermission checks).

Common Services (CS) User Preferences

• Current usage
  CS stores user preferences (e.g., theme, timezone, module order). MFEs read these preferences on login via CS APIs. UI already fetches preferences.

• Product ask
  Allow updating user preferences from within TSP uplift UI (Payments MFE) using CS “user preferences POST” API. CS owns DB and APIs; we would invoke CS to update only uplift-related keys.

• Team’s considerations and stance
  • Preferred: Users update preferences in CS portal; MFEs only read. Avoid cross-app writes to prevent accidental overwrites of other apps’ preferences.
  • If mandated: Restrict updates to uplift-specific keys only; do not send or mutate unrelated ITC/GSH preferences. Validate CS update API payload granularity/merge semantics.
  • Security and scope: Ensure only uplift-scoped keys are sent/updated. Confirm CS supports partial updates and does not require full object replacement.

Environment and Deployment Notes

• Deployments
  • UI MFEs are on GKP; TSP Web currently on VM but accessible via gateway routing to provided hostnames.
  • Payment Gateway is deployed in Dev GKP.
  • Target is to progressively move services (including TSP Web) to GKP and consolidate behind one gateway.

• Testing blockers
  Enable test users in Recipient Store with proper entitlements and document required headers/claims. Update Swagger to include auth schemes for accurate testing via tools (Swagger UI/Bruno/Postman).

📋Overview

• Payment flows: All MFEs load via CS routes and call backends through the Payment Gateway; TSP Web currently pass-through, Recipient Management validated at gateway.
• Gateway policy: Temporary filtration bypass for TSP Web will be removed after TSP migrates to shared controls on GKP; aim for single DSP gateway across systems.
• Recipient Store: Required to externalize recipient data; read-only integration first (search, get-by-id). Calls ideally through our gateway; internal calls via service mesh.
• Auth/onboarding gaps: 401/403 issues indicate missing bearer token handling in docs and lack of user onboarding/entitlements in Recipient Store. Need alignment on entitlement model (simplified vs legacy).
• CS Preferences: Read is in place. Product wants in-app update; team prefers CS-portal-only updates. If proceeding, restrict to uplift-only keys and verify partial update semantics.
