📒Meeting Outline

CodeForce: Standardized AI Engineering Agent for Frontend (Phase 1) and Full-Stack (Planned)

• Problem with prompt-driven agents and rationale Current AI/copilot tools rely heavily on ad-hoc prompts, leading to inconsistent outputs and frequent prompt rewrites. CodeForce aims to remove prompt dependency by standardizing inputs via user story/feature IDs and enforcing a structured SDLC.

• End-to-end capability scope CodeForce adds integration, security, accessibility, marketing support, greenfield/brownfield handling, automated unit testing/coverage, and build. It produces production-ready documentation, performs architecture review, design, coding, security checks, accessibility conformance, testing, and build handoff.

• Input model: Rally-backed automation Instead of a free-form prompt, users provide a Rally user story/feature ID. CodeForce fetches description, acceptance criteria, attachments, and creates a PRD (product requirements document) and development plan before generating code.

• Security and compliance posture The agent is trained to avoid downloading artifacts outside the approved environment (“opt-in” perimeter). It ensures security compliance, reuses existing architecture/components for brownfield apps, and avoids installing new external libraries unless creating a greenfield app.

• Accessibility and UI library strategy Enforces accessibility for new screens and supports multiple UI libraries/themes (e.g., Skyline, Material UI, Harmony) based on user requirements and org standards.

• Brownfield vs. Greenfield handling

  • Brownfield: inspects existing architecture/components/libraries, selects approved dependencies, writes code to fit the stack.

  • Greenfield: bootstraps full application setup from scratch, including project scaffolding.

• Testing and build reliability loop Auto-generates unit tests for new functionality, runs all existing tests, fixes failures, builds artifacts, and hands off only after a successful build.

• Current flow and future roadmap

  • Current: requirements intake → analysis → architecture → UI generation → validation → security → accessibility → test execution.

  • Roadmap: add backend .NET agent (in testing), evolve into a full-stack agent completing frontend, backend, and integration from a single user story; future “bot” mode to auto-claim Rally stories assigned to “CodeForce,” implement, and raise PRs.

• Performance benchmarks Internal pilots (OWCA, Automatics): end-to-end CRUD screen (incl. tests, security, analysis, review) reduced from ~7 hours with generic AI tools to ~3 hours with CodeForce (~50% effort reduction). Sample timelines shared for 1/5/10 screens.

• Access and distribution Delivered via organization-level GitHub integration (OWCA). Appears in VS Code’s Agents dropdown for entitled users. Version updates auto-propagate. No additional licenses. Phase 1 covers UI agent + Rally integration; next adds .NET backend; later a Rally bot.

• Live demo highlights (Rahul)

  • VS Code workflow: select CodeForce agent, enter user story ID, agent installs necessary tooling (e.g., Playwright for Rally SSO), prompts for using current repo as UI workspace and backend path (frontend-only in Phase 1), initiates SSO via browser, fetches Rally details.

  • PRD Builder sub-agent compiles an MD-based PRD with business context, scope, description, attachments.

  • Agent proceeds stepwise; asks clarifications if build issues arise; generates code, unit tests, and a final report summarizing screen location, chosen UI library (e.g., Skyline), theming, test artifacts.

  • Output demo: generated React screen adhering to Skyline styling/branding, with mock data when backend is declined; edit and extend via the same agent prompt thereafter.

• Security/SSO Q&A (Manohar)

  • Authentication occurs through web SSO in the user’s browser. No hardcoded credentials; redirects to MSID login where applicable. Only upon successful verification does it fetch Rally artifacts.

  • Rally data retrieval uses a Python script post-SSO to generate an API key scoped to the user/team and fetch details into VS Code.

• Adoption and standardization discussion (Shahbaz and response) Recognition for innovation; emphasis on scaling, positioning, and standardization to avoid duplicative efforts. Move from curiosity phase to maturity: consolidate practices, enterprise-grade frameworks, and reuse across teams. Consider aligning with Tiger Team initiative for platform-level leverage and broader recognition.

• Guidance on user story quality (Abhijit’s query) More detailed user stories yield higher accuracy. Comprehensive business functionality, clear acceptance criteria, and attachments improve the PRD and downstream code generation. The agent surfaces the generated MD document for review prior to development.

DMA Data Engineering Bot (DEB): AI-Augmented ETL SDLC for IDW

• Context and goals Data engineering faces rising complexity: multiple sources, tighter timelines, higher quality bars, and context switching. DEB augments (not replaces) the existing architecture/standards to automate heavy-lift steps for batch ingestion first, expanding to other patterns.

• Current (manual) DE SDLC baseline Convert business user story into a mapping spec (target schema, NKs, columns, transformation rules), handcraft unit tests, configure the DXF accelerator (table-level configs for sources, transforms, SCD/keys/encryption), orchestrate pipelines, run loads, execute tests, open PRs, and promote through environments—largely manual.

• Target AI-enabled flow with human-in-the-loop

  • Inputs: Business user story transformed into an AI-ready user story (ARUS) with required sections; completeness validated by a “user story validator” (yellow = pending module).

  • DEB planner generates an execution plan from ARUS, including code/config creation, job/pipeline setup, execution, and unit testing steps.

  • Build agent creates:

    • JSON configuration for DXF accelerator (source/target/transforms/keys/partitions/perf).

    • Databricks jobs/workflows with cluster, tasks, dependencies.

    • Validation notebooks and unit tests from the mapping spec.

  • Execution staged initially in a separate AI environment for confidence, then mirrored to dev by moving code and re-running validations.

  • Developers review at gates (plan approval, code/config sanity, execution approval, merge oversight). Future agents will add validation/deployment automation.

• Architecture components

  • Developer UI to interact with DEB.

  • Modular agents: planner and build (current), validation and deployment (planned).

  • LLM: Claude Sonnet via Claude SDK.

  • Knowledge base (RAG) built with Microsoft HV: HPE Core Researcher converts the framework/accelerator codebase into structured English MD modules (dimensions, facts, stages, inputs), enabling effective chunking and retrieval.

  • Databricks integration via MCP and Databricks skills for job, config, and notebook operations.

• Demo highlights (Bharat)

  • ARUS hosted in Git issues (Rally planned later), including acceptance criteria and mapping spec.

  • Planning phase outputs a JSON config preview and enumerates next steps (create/upload JSON, update unit-test workflow, create validation notebook/tests, execute tests).

  • Build/execution:

    • JSON config generated with source/target and transformation rules.

    • Unit-test job added/updated in Databricks with compute and task configuration.

    • Pipeline executes to load target tables (bronze → gold per medallion).

    • Validation notebook auto-generated from mapping spec covers nulls, PKs, transformations; tests executed with results summarized and links provided.

  • Efficiency assumption: conservative 50% time reduction initially; to be revisited post-adoption.

• Rollout plan Pilot with two scrum teams in DMA, expand to others. Currently local; to be deployed via UIS and AIR processes. Longer-term automation to cover remaining gray-box steps.

• Q&A on knowledge base creation HPE Core Researcher agent ingests code and emits MD-formatted English documentation of modules, stages, and inputs; this serves as the RAG knowledge base for DEB. This approach outperforms direct code chunking for retrieval and reasoning. Alternatives (e.g., other code-to-doc approaches) exist, but HV Researcher yielded strong results.
