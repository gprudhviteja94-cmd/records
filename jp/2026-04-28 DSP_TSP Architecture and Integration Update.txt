📒Meeting Outline

DSP/TSP Architecture, Scope, and Integrations

• Design confirmation and API scope
  Team confirmed the new “discipline management” design and related services are standalone. All endpoints are new and not integrated with existing DSP hub logic. Objective is clarity on whether any existing APIs are reused—answer: no reuse; standalone services.

• TSP application overview and supported products
  TSP handles payment initiation “high value” (wires), real-time payments, wallet payments, and newer consent/mandate management (non-financial messages). Low-value ACH support has begun with India ACH; redesign underway to support global ACH. External/vendor-initiated payments for tax/utilities are supported via TSP access.

• Workflow and approvals
  All pay types share a common workflow in TSP except mandate/consent flows, which differ slightly. Client-configured, multi-level approvals are supported. Roles and entitlements are maintained in Admin Center.

Entitlements, Roles, and Decommissioning Plan

• Current roles integration path
  TSP consumes roles via WPI Roles Adapter, which then calls Admin Center Roles API. This detour exists for historical reasons.

• Runtime (RT) database decommission
  The “runtime” database currently stores certain entitlements. It is scheduled for decommissioning around June–July. Strategy: migrate entitlement reads from RT to Admin Center/ROSE entitlements directly, eliminating WPI detours over time.

• Scale and connectivity
  Approximately 196K clients and 196K users are entitled. There are 50+ upstream/downstream interfaces.

Templates, Beneficiaries, and File Import

• Payment templates and mass payouts
  Templates can prestore multiple beneficiaries to enable one-click initiation of many payments. Current TSP supports this via templates; uplift program plans a “Benny-first” global feature to streamline beneficiary-first flows.

• File-based initiation
  Two file import paths: (1) direct upload via UI (supporting two file formats), and (2) client uploads to S2H; S2H forwards to TSP for processing through the standard workflow. Users must follow a defined upload process before extraction.

• Resiliency and support tools
  TSP Support app is being technology-upgraded and functionally rewritten (new UI and backend) to assist production support with configuration and resiliency.

Upstream/Downstream Architecture and Market Nuances

• Reference/static data and messaging
  TSP is upstream-connected to PBRM for static/reference data needed for market onboarding and new pay types. One-way publish via Kafka for certain data; two-way interactions exist where required.

• Market coverage and routing
  6–7 markets (LATAM/APAC) currently flow via the Kafka-based route; 45+ other markets follow legacy paths. Market-specific exceptions (e.g., Brazil, Russia) use direct connectivity. China routing migrated from CCAP to WPI last year.

• Back office and routing components
  Post-approval, payments route via WPI to strategic back offices under the “Graphite” umbrella (Graphite and Plutus). RTP vs other payment types may follow distinct processing paths before acknowledgments/callbacks return to TSP for status updates.

Callbacks, APIs vs Queues, and Notifications

• Callbacks and transport choices
  WPI returns acknowledgments/callbacks to TSP using TSP-exposed endpoints; WPI prefers messaging queues for scale/reliability. APIs are used where immediate responses are necessary; otherwise, queues are preferred for extraction and status.

• Notifications
  TSP and WPI publish processing statuses (primarily from WPI) to Kafka; common services consume and surface these as UI notifications within TSP.
