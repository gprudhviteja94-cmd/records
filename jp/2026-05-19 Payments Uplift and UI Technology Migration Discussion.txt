📒Meeting Outline

Payments Uplift and UI Technology Migration

• Uplift scope and objectives
  The uplift is primarily client-focused, targeting improved batching capacity and a better UI experience (Danujan product). Aim is to support up to 100,000 payments per batch so complete GSH can flow into DSP.

• Technology migration (Ember to React)
  Current UI is on Ember; target is React. Request to add a “Technology Current” column in documentation to clearly indicate stack changes (e.g., Ember → React) for each flow/segment.

• UI isolation and navigation changes
  Flows will be isolated with a clearer view; mega menu/navigation remains but is refined. Movement to “Standard” navigation persists; backend behavior remains unchanged.

Batch Limits, Market Needs, and Constraints

• Current limits and regional asks
  Present UI limit is around 100 payments per batch. There is an active request from Saudi and other regions to support 500,000 payments per batch to avoid repetitive operations (e.g., paying 500 subcontractors requires multiple runs today).

• Root-cause investigation (UI vs server)
  Need to determine whether limits are due to UI real estate, server constraints, export mechanisms, or other factors. Initial view: some constraints (e.g., Excel/PDF exports) likely not UI-related.

• Operational impact and client workflows
  Monthly use cases require separate templates per company; current 50–100 item constraints force 5–10 repeated operations for larger payee sets. Expanded batching would materially reduce operational burden.

Exports, Templates, and Data Handling

• Export limitations (Excel/PDF/media)
  Users analyzing CDPR are unable to download ≥1,000 records (often capped around 250). This likely ties to backend/export services rather than UI. Need to review Excel generation and media limits.

• Template management
  Separate templates per company are needed; larger batch operations should align with export and import capabilities to avoid bottlenecks.

Backend Integrity and Feature Toggles

• Backend impact assessment
  Backend code and data contracts remain unchanged; uplift focuses on UI rendering (“painting the screen”). Payment workflows and FDM-related logic remain intact.

• Deployment philosophy and toggles
  Team historically leans toward confident “all-in” deployments without toggles (cited prior success). Discussion on whether to enable selective rollout/toggles, especially for U.S. Wire scenarios, to support limited-client enablement if needed.

U.S. Wire and Rollout Readiness

• U.S. Wire context
  Importance highlighted for U.S. users; any enablement for even 1–2 clients should include clear support/escalation (ping responsible owners).

• Readiness and stability
  Team feels prepared; no expectation of shutdown or disruption. Opportunity to onboard some clients on M1/standalone uplift without broad production toggling, if feasible.

📋Overview

• Uplift will enhance client UI and batching to target 100,000 payments; full GSH to DSP integration is a goal.
• Migration from Ember to React proceeds; add “Technology Current” column to track stack changes per flow/segment.
• Batch and export limits hinder large clients; need to distinguish UI vs backend constraints and plan scalability.
• Export caps (Excel/PDF/media) affecting CDPR analysis likely stem from backend/export service limits, not UI.
• Backend contracts remain stable; uplift is UI-layer only.
• Consider pragmatic feature toggles for controlled rollout, particularly for U.S. Wire, despite prior all-in success.
• Team is confident in readiness; navigation shifts to Standard persist without backend impact.
