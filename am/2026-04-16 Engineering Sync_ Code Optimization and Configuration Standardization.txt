Code optimization: central utility for configuration values

• Problem: repeated code for IDs
  The function to assemble input parameters (use case project ID, platform ID, dataset IDs, DW, etc.) has been duplicated across many locations, leading to inconsistency and maintenance overhead.
• Solution: utility method with dictionary config
  A utility method now accepts an added parameter: dictionary config. Values are read from environment configuration and injected into input parameters (inputParam.withConfig), removing repeated snippets like inputParam.set(key, ...). Callers can retrieve only the needed attributes while using the same method.
• Adoption guidance
  Going forward, avoid writing repeated ID/config code. Reuse the utility to fetch required attributes. This will standardize behavior and reduce errors across new use cases.

Environment variables and configuration standardization

• Klumi environment variables consolidation
  Dynamic parameters will be fetched from environment variables. A cleanup is planned to model project variables as a single JSON object within the environment for direct property access, reducing custom JSON mapping overhead in code.
• Task-level vs. global availability
  Environment-derived values will be globally available. Best practice: set shared values into context data in the first task, then read from context in subsequent tasks. Per-task retrieval remains possible when needed.
• Scope limits
  Task-specific parameters in QTrack configuration still need to remain within that mapping; only globally repeatable parameters will be moved to environment-sourced configuration.

Data storage and payload size considerations

• Concern on case JSON size
  Seven raised concerns about growing JSON size. While ECM (as a reduced cassette) could be an alternative to DB for certain payloads, limits must be confirmed with the ECM team (max JSON size).
• Current decision and rationale
  Maintain current database-based approach (previously reviewed and approved by Raj) until a clearly feasible alternative (e.g., Redis/ECM with confirmed limits) is validated. The new environment/config changes should already reduce JSON payload sizes.
