📒Meeting Outline

Version handling for RPT model (API, UI, and data model)

• API strategy (single vs multiple calls)
  • Team discussed avoiding a separate API just to fetch the current version. Proposal: the primary RPT load/details API should return both values in one response: the selected/version-at-creation and the current system version.
  • Agreement to combine backend logic so the load/details API includes: version (selected/active) and base/original version, plus currentVersion for the platform.

• Version definitions and fields
  • Standardized field semantics:
    • version (a.k.a. selected version): the active version user is working on and that gets updated on autosave when user switches.
    • baseVersion (a.k.a. original version): the version from which the RPT was duplicated/created; immutable and never changes.
    • currentVersion: the latest available platform/model version; used as an option in the dropdown.
  • Naming to be finalized (baseVersion vs originalVersion), but behavior is confirmed.

• Dropdown behavior and options
  • On initial load: show two options only—baseVersion and currentVersion.
  • Default selection is version from the RPT model (initially equals baseVersion on first load after duplication).
  • After switching to currentVersion and autosave, UI refreshes with updated model and version equals currentVersion.
  • If version equals currentVersion post-switch, dropdown still shows two options: baseVersion and currentVersion (so user can revert to base). No historical “last used” or intermediate versions are kept.

• Autosave and UI refresh
  • When user selects currentVersion, trigger autosave; backend returns updated model JSON reflecting the new version schema (including any new fields).
  • Front-end re-renders the UI on the same page using the returned model; selected version persists as version; baseVersion remains unchanged.

• Data persistence and JSON model
  • Autosave response must return the full updated model JSON specific to the RPT (not just a diff or reliance on generic version configuration).
  • Store and honor both version and baseVersion in every response/save cycle. baseVersion is read-only; version is writable based on user selection.
  • Ensure newly introduced fields from higher versions are included in the model JSON even if not immediately filled by the user.

• Scope and constraints
  • Only two dropdown options are ever presented: baseVersion and currentVersion.
  • No support for stepping through intermediate versions (e.g., V3, V5) or maintaining “last used” version state.
  • Switching back and forth is allowed only while the RPT is in Draft state.

Edge cases and defaults

• Newly duplicated RPT with no edits
  • version defaults to baseVersion (the duplication source).
  • Dropdown still shows baseVersion and currentVersion; no autosave occurs until the user switches.

• Unavailable dropdown in certain environments
  • Noted that deployed/stage UI discrepancies currently hide the dropdown; once environments align, dropdown will reflect baseVersion and currentVersion properly.

• Post-switch equality scenario
  • Even when version equals currentVersion after autosave, retain dropdown with both baseVersion and currentVersion so user can revert to base while in Draft.

📋Overview

• Adopt a single RPT load/details API that returns version (selected), baseVersion (immutable), and currentVersion.
• Maintain baseVersion permanently as the original duplication version; never mutate it.
• Treat version as the active/selected version; update it on autosave when user switches in the dropdown.
• Dropdown always shows exactly two options: baseVersion and currentVersion; no intermediate or “last used” tracking.
• On version change, autosave must return a fully updated RPT model JSON reflecting the selected version’s schema; front-end re-renders accordingly.
• Switching between baseVersion and currentVersion is allowed only in Draft state.
• Default selection on first load after duplication is baseVersion; no selection memory beyond version field is needed.
