📒Meeting Outline

Automation of ACDV Dispute Handling and Validation Flow

• Eligibility and account pattern checks
  Business rules determine dispute eligibility based on account number patterns (e.g., empty, starts with 61, lengths 13/16 invalid), and disallowed patterns. Ineligible disputes are business-exceptioned before further processing.

• GCBR report and request construction
  System builds the search request, fetches account and override details from GCBR, and constructs the GCBR report used to query Triumph.

• Data retrieval from Triumph and GAR
  Triumph retrieves GAR account master details and performs early ACDV validations; system also retrieves account balances from GAR, identifying usable balance types (e.g., unbilled). Any bundle exceptions lead to business exceptioning, potentially skipping CLiC case creation based on conditions.

• Negative/deletion policy checks (pre-deep verification)
  Applies negative/deletion policies before deeper checks: evaluates dispute code (e.g., 111), account canceled status, SCC (special comment code), date of first delinquency (DOFD), line-of-business rules, and status-driven validations. Failing items are business-exceptioned.

• Exception filtering updates (sold accounts)
  Exception filter expanded to include SCC “AH” (sold accounts). Previously, only specific brands (JetBlue, Costco) with AH were filtered; now all AH are filtered alongside existing A, B, C, CO.

• RPA-based account status and history retrieval
  RPA retrieves account status, plastic history, and delinquency data. RPA failures are caught and exceptioned.

• Satisfactory deletion (7-year) check
  For negative request scenarios, compares Triumph FF30 cancellation date vs. current date. If >7 years, business-exceptioned; otherwise proceed and flag set to continue ACDV closure.

• Second-pass exception filtering
  After receiving additional Triumph details and applying satisfactory deletion checks, remaining exception scenarios are filtered again to ensure only valid cases proceed.

• Bankruptcy details via ACON
  Converts account number to account token with ACON utility; fetches last-stage collection/bankruptcy details and maps to card/loan accounts when the earlier-set condition triggers this path.

• Consumer information verification (e-OSCAR)
  Verifies ACDV consumer section: DOP, SSN, ECOA, PHP, and all consumer fields; marks each as same/changed; retrieves credit limit. Care enrollment check is performed.

• Account information verification (e-OSCAR)
  Builds account info view, evaluates account status, payment rating, CCC, SCC, portfolio type/date, etc., marking fields same/changed.

• Response code computation and submission
  Computes ACDV response code based on dispute context and field changes (e.g., 01 for no change; 21/22/23 for specific field-change scenarios). Creates CLiC note for closure rationale; validates response via APIs; submits final ACDV response to e-OSCAR. Validity of ACDV is checked prior to submission.

Business Perspective and Use Case

• End-to-end functional intent
  Automates the bureau analyst workflow: aggregates data from multiple SORs (GAR, C360, ACON, Triumph, GCBR), verifies consumer and account info against e-OSCAR ACDV, determines if and what fields changed, responds back to ACDV with appropriate codes, and documents rationale in CLiC.

• Operational improvements
  Reduces manual verification effort, standardizes rule application (eligibility, negative/deletion, bankruptcy), and tightens exception handling for edge scenarios (e.g., AH sold accounts).
