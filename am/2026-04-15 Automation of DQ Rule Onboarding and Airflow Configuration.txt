📒Meeting Outline

Automation for DQ Rule Onboarding

• Current manual process and pain points
  The business submits DQ rule requests (table, column, check type, description, schedule). Devs manually draft SQL based on similar existing rules, then update Airflow DAG mapping and QTrak config JSON; rules execute only when present in DAG. There are ~1500–1600 rules across 300–400 tables, grouped under five check types (duplicate, missing value, stagnant, threshold, valid value). Threshold is largest (~900–1000). Manual process is slow and error-prone; a single JSON change blocks parallel PRs.

• Self-service automation scope (Task 2)
  On new entries in DQ_repository, the job generates SQL from templates and updates the appropriate DAG (daily/weekly/monthly) by appending to existing table-level SQL files; then raises a PR for merge to prod after testing (E1→E2→E3). A “status” column (pending/in-progress/implemented) in DQ_repository is requested to drive idempotent processing.

• Template strategy and deviations
  Many rules share identical descriptions; these are clustered into templates. Four of five check types templated; threshold partially covered (22 templates planned; ~10% ready). If a new request doesn’t fit any template, it gets routed to a developer for custom SQL; future new patterns become new templates.

Airflow, Config and Code Review

• Airflow execution and configuration
  Each rule must exist in a DAG file to run. DAGs and SQLs currently reside in GCS under Composer; plan is to move SQLs out of Composer storage to a project use-case bucket. QTrak config JSON maps rules/DAGs/schedules. Airflow lists DAGs by scanning Python files with airflow decorators from GCS; run metadata is stored in Airflow’s backend DB.

• Code structure and review highlights
  Entry: sql_generation_dag.py with taskflow pattern; an executor orchestrates steps: load config, read input rows from BigQuery, generate SQL from templates, save results, update legacy SQL, generate reports, update output folder.
  • Naming fix: “initialize_bigquery” actually loads configuration, not a DB connection. Action: rename to “initialize_config” to avoid confusion.
  • Ordering/efficiency: Don’t initialize BigQuery client before validations; minimize early resource use. Consider a finally block/task to clean up context on failures.
  • Env configs: deploy-config file missing; testing had been manual uploads. Action: standardize deployment via Ayyappa; no manual Composer uploads.
  • Parallel processing: Consolidated into single DAG taskflow to pass context without XCom per Nirmal’s guidance.

• Repository and PR hygiene
  Keep feature branches; PR to develop only after review sign-off. Avoid PRs to main. Local-testing folder exists but excluded from review per Sachin; focus on deployable Airflow code.

Data Source, Storage and Scheduling

• DQ request intake
  Front-end to replace email intake; will write to DQ_repository (BigQuery). Automation polls for status=pending requests. Filter criteria column must be incorporated alongside description/check type.

• SQL and schedule mapping
  A single SQL file can contain multiple queries for a table; new rules append to existing file based on schedule (daily/weekly/monthly cron). Jobs should extend existing artifacts, not proliferate files.

• Storage location policy
  SQLs should not live in Composer storage; move to use-case GCS bucket. Only executables should sit in Composer.

Alternatives and Architecture Improvements

• DB-backed configuration vs JSON
  Sravan proposed storing DQ configs in BigQuery tables to avoid large monolithic JSON and PR conflicts; enable concurrent updates and reduce blast radius. Options discussed:
  • Direct Airflow read from DB: not feasible without altering company-wide Composer behavior.
  • Compromise: maintain authoritative configs in BigQuery; CI/CD job periodically materializes a master JSON into GCS for Airflow to consume. This allows parallel edits, auditability, and reduced merge contention.
  Sachin cautioned about increased runtime DB connections and latency; current JSON is deployed at E3 and cached by environment. Further design review pending.

• Containerized frameworks
  Sravan described prior pattern: Airflow DAGs trigger lightweight container images (DQ/ETL) that encapsulate connections, execute, emit results, then exit—keeping runtime lean.

• Agentic/LLM-based SQL generation
  Prithvi presented a POC concept: few-shot, template-free SQL generation and validation using internal LLMs/agents with feedback loops and environment validation. Constraints: strict LLM access; metadata sensitivity; running queries against prod/test may expose sensitive data. Not greenlit; could be reconsidered with strong use cases and internal LLM allowances.

Governance, Security, and Platform Constraints

• Open-source restrictions
  MX prohibits external open-source dependencies and outbound network access; internal curated libraries only.

• Security and cost
  Limit BigQuery connections and unnecessary resource initialization; avoid Composer storage misuse; formalize cleanup on failure paths. LLM usage constrained due to data sensitivity; internal offerings may arrive later.
