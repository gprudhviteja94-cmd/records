New framework guideline: configuration-driven inserts/updates (no new Python files)

• Config-first task design
  • For any insert (and update) operations, do not create new tags or Python files. Use a generic task in the framework file with SQL kept in configured paths; execution handled by existing utility methods.
• Utility-centric input handling
  • All repeatable input processing is centralized in utility methods. Input params to be registered in the utility; tasks must avoid direct input object mutations (e.g., no input.update patterns). Use get_attribute/execute and new insert-data execution helpers.
• XCom and parameter management
  • XCom variables validated via utility (validate input param). Teams can register new XComs/conditions as needed without changing structure.
• Backward-compatible configuration structure
  • Maintain prior config layout: properties, XComs, SQL file paths. Single or multiple tasks can be defined; parameters maintained in input params. Implement similarly for update paths.
• Scope
  • Applies uniformly across environments (E1, E3, etc.). All owners working on update/insert parts should adopt immediately.
