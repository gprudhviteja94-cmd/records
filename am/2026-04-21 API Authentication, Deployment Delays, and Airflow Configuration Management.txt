Airflow DAG/config management for lower environments

• Scope of config changes (QTrak/QTrack config JSON)
  The file in question is QTrak/QTrack config JSON (qtrak_config.json). The proposal is to avoid copying the entire large file (with ~9000+ lines and many DAG objects). Instead, keep 1–4 representative JSON objects and append only new keys or DQIDs needed for testing.
• Avoid full-file copy and excessive DAG registration
  Copying the full config into the environment would auto-register many DAGs in Airflow, potentially overloading the lower environment. Prior incident: CPU usage spiked, requiring termination of DAGs. Guidance: do not bombard the lower environment; keep only minimal required JSON objects.
• File naming and deployment behavior
  Keep the same filename (qtrak_config.json) to avoid additional code changes post-deployment. Using alternate names (e.g., *_test.json or .txt) is discouraged for deployment paths. Airflow detects changes and overlaps/reloads DAG definitions on redeploy; duplicating the same DAGs will overlap rather than register new ones, but large configs still create load.
• Working approach for new application (Automation Service)
  For the new Automation Service folder structure:
  • You may copy qtrak_config.json locally but prune it to only a couple of JSON objects (1–4) relevant to your checks.
  • Modify/add only the required DQIDs (e.g., existing DQID 9129; add DQID 12) without creating new DAGs.
  • If multiple check types exist, include only those necessary JSONs to support the logic; otherwise, use one template JSON and adjust DQID/check-type parameters.
• Table creation ownership
  If a table is requested, it should be created by the requester via their code under the temp folder structure only; do not create use-case buckets or top-level use-case folders.
