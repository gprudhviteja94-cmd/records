📒Meeting Outline

CL Verification Agent: Problem Context and Goals

• Current SMA verification workflow and pain points
  Today, seal/CL verification is manual (via SMS/notes), requiring SMEs to pull data from multiple tools (Critic, Buganizer/Organizer, Test Tracker, Rapid) and read keynotes/CL code changes. They review past defects and experience to decide if integration testing is required and what to test. This process is time-consuming and error-prone.

• Objective of the CL Verification Agent
  Automate consolidation, reasoning, and test recommendation to determine whether manual verification/testing is needed for a given CL; reduce SME effort and latency while improving consistency.

• Scope and scalability considerations
  Initial focus on Gmail Web with intent to scale to Chat, Voice, and Spaces (Workspace). Agent to integrate later into a broader Test Execution Orchestrator; CL verification will function as a sub-agent.

Architecture and Data Flow

• Agent and sub-agent design
  A root CL agent orchestrates sub-agents to fetch data, reason over it, and produce a verdict with rationale. Fetch stages use tool APIs; reasoning stages use LLM-based analysis.

• Systems integrated and retrieval details
  APIs used to gather data from Critic, Buganizer/Organizer, Rapid, and Test Tracker. CL-level details and interconnected CLs are examined; historical runs and outcomes are stored in RAD for semantic search and learning.

• Reasoning, recommendations, and outputs
  The agent:
  • Consolidates CL changes, related CLs/bugs, and test assets.
  • Performs LLM reasoning to assess impact and derive a verdict (e.g., “manual verification required” or “no manual verification required”).
  • Retrieves similar historical CLs via semantic search in RAD, along with prior QA notes/outcomes.
  • Recommends prioritized tests from Test Tracker mapped to the CL description and context.

Privacy, Compliance, and Data Retention

• Outcome of privacy/compliance consultation
  Storing bug descriptions in RAD is not permitted. Pivot made to store CL details/descriptions and test run history instead. Team to confirm this approach is aligned with AI policy/compliance teams in writing.

• Traceability considerations
  While bug IDs help traceability, storage is constrained; current approach emphasizes CL-to-test correlation. Team will verify policy alignment and document allowed fields.

• User-facing experience
  Final UX will be a plain chat-style interface; current UI is for testing/demo only. Token/time metrics are captured internally for performance monitoring.

Demo Highlights and Performance

• Demo setup and execution
  • Ability to filter a CL range (e.g., last two quarters), fetch CLs, and run analysis individually or in batch.
  • Run status shown (Pending/Completed); exportable CSV report for multi-CL runs.

• Sample outcomes and rationale
  • Example 1: “No manual verification required” for cleanup/logging/parameter naming changes; detailed tool-level reasoning provided.
  • Example 2: “Manual verification required” for front-end look/UX changes; recommended tests listed with references.

• Performance metrics
  Typical run time ~33–120 seconds; some cases up to ~4 minutes. Time is non-blocking for users. Tokens/time recorded per stage.

Learning Loop, Persistence, and Gaps

• Human-in-the-loop feedback
  Users can Accept/Correct the agent’s verdict and recommendations; feedback stored in RAD to improve future reasoning for similar CLs.

• Session/memory behavior
  Currently, if a CL run is not “accepted/committed,” it may reappear as pending on refresh; final deployment will persist run state and acceptance to avoid rework.

• UAT/accuracy status
  Ongoing UAT over 4–5 weeks across 100+ daily CLs to compare agent output with human decisions. Reported accuracy is improving; discrepancies stem from human exclusions vs. agent inclusions. Further tuning in progress.

  • Medium term: Start with Gmail Web; after a successful trial, expand to other platforms (Voice, Chat, Spaces).
