📒Meeting Outline

GoldenGate adoption and stakeholder support

• Project context and support channels
  The team is actively developing and deploying agent solutions and facing challenges with GoldenGate framework setup. Abhimanyu, the L300 team, and members from GDC-side offered hands-on support and escalation guidance.

• Stakeholder identification (GBM/FTE)
  Emphasis on identifying the correct Google Business Manager (GBM) / FTE stakeholders who can approve IAM, billing, and access. Many delays come from not knowing the right approver; use IAM visibility to find owners. Expect 1–3 days for GBM/FTE to understand scope and roles.

• Parallelization and sequencing
  Several prerequisites can proceed in parallel, but some must precede others (e.g., billing before Vertex/Gemini API). Leads/managers should start critical steps pre–Day 1 to avoid multi-week delays.

Prerequisites: Projects, billing, groups, and access

• GCP project and billing association
  A GCP project with an associated billing account is mandatory, especially for Vertex AI/Gemini (no free tier). Billing enables API keys; without it, agent dev is blocked. Share step-by-step docs to non-technical PMs to avoid creation delays.

• Org components and budget (Organizer)
  Each project needs an Organizer/Buganizer component ID and budget component. If missing, request creation via project owners/GBM. Associate relevant Google Groups during project setup.

• Lantern/JFG Next groups and roles
  Set up JFG groups via Lantern/JFG Next per project need. Define group names and roles; GBM should propose the JFG group structure. Provide process references for TBCs and align with vendor managers.

• Strings access and business justification
  Use Strings to manage policy-driven access. Prepare solid business cases for unrestricted or sensitive accesses; approvals are typically quick once rationale is clear.

• Production path enablement (G3 authors space)
  For productionization, GBM must enable the production path in G3/authors. Expect 1–2 days for path/team approvals; align IAM groups (MDB) and policies early.

• Additional tool/project-level accesses
  Plan 2–3 weeks buffer for extra tool/API accesses (Organizer, Sponge, trackers, etc.). Billing permissions are required before API keys are issued. Document expirations and blockers.

Technical design documentation (Elephant–Owl–Goldfish)

• EOG document purpose and flow
  Before production, prepare Technical Design Document following Elephant (context), Owl (analysis/statefulness), Goldfish (validate/critique) structure. Include a mandatory architecture diagram and list approved tools/APIs.

• Tooling recommendations and exceptions
  SpannerDB is recommended; Cloud SQL may be used with exception approvals. External APIs (e.g., Zendesk, GEL test APIs) typically require exceptions. Using recommended tools speeds approvals.

• References and samples
  Team will share a masked, approved TDD example and internal talk link to guide format and content.

Reference architecture and data/storage choices

• SpannerDB recommendation
  GoldenGate recommends Spanner for security and internal compliance. It’s not MySQL/Cloud SQL; schema under review with FTEs is pending. Access is tightly controlled—changes should be code-driven and reviewed.

• Block store for large artifacts
  For multi-GB/TB payloads, use approved block store services for instant retrieval and managed spaces; integrate with agent IO paths.

• Queues for long-running tasks
  For jobs >10 minutes (beyond RPC timeouts), use SparrowQube (preferred for easier security approval) or Manyfold (more customization) per use case.

UI and user experience constraints

• Supported UI patterns
  Minimal UIs via Gemini Enterprise chat are currently preferred/allowed. Rich UIs (Angular/BoxWebline/BoxCX) face stringent RPC/stubby connectivity approvals and long lead times; only one team reportedly obtained SSO so far.

• Approval challenges
  UI-to-backend RPC is the main bottleneck due to strict code-leakage prevention and stubby policies. Expect months to obtain necessary RPC pathways; plan for CLI/chat interfaces where feasible.

Access, approvals, and process accelerators

• FTE dependency and escalation
  Many steps require FTE action (latchkey confirmation, IAM CL reviews, GoldenGate CLI execution, agent registration). Prior projects took months until the right approvers were identified; recent cycles reduced to ~2 weeks with known contacts.

• Knowledge re-use
  Leverage known proactive internal teams and named contacts (to be shared offline) to accelerate Spanner, GoldenGate, and policy approvals. Ask managers to include these approvers early.

• Security/data policies and guardrails
  Include data security/PII policies in design and SOW: pre-LLM guardrails (masking before send), post-LLM remapping, sensitive data handling, and required CL-reviewed controls.

Latchkey, IAM policy, and CL workflow

• Latchkey enablement
  Latchkey (infra-as-code for GCP in Piper) is implicitly enabled when billing is associated. If unsure, get owner confirmation. Create IAM policy YAML listing roles/members (MDB/Google groups) using provided templates; submit as CL for FTE review.

• Policy review cadence
  Expect iterative review cycles (LGTM/LWT); keep approvers engaged. Document owners, groups, and role bindings clearly to minimize back-and-forth.

GoldenGate CLI, authors migration, and dependencies

• GoldenGate CLI usage
  Use GoldenGate CLI to scaffold and deploy agent services to Box servers. FD involvement is mandatory. Buganizer component ID is required for steps and tickets.

• Authors environment changes
  Secrets and clients differ in authors space; replace external client libs with first-party internal equivalents. Local testing via orca slipper; production path follows GoldenGate hierarchy.

• Service/RPC compatibility
  Some external tools (e.g., Vertex AI Search) may be incompatible with authors; switch to internal alternatives (e.g., MoMA Search) with team approvals and RPC permissions.

Agent registration and publishing

• Registration via GBM/FTE
  Manager/FTE must file the registration request detailing retention, service connections, leads/operations, and code review. Create a Buganizer ticket using the component ID; include agent card and values from gate.yaml.

• Dolphin server configuration
  Add dependencies to allow Dolphin server to connect to the agent; include agent card and page address. Create and submit CLs; engage FTE reviewers for LGTM/LWT approvals.

• Elder/final production stamp
  Moving from POP (GoldenGate namespace) to agent space requires Elder/production stamp—a lengthy, security-heavy process that can escalate to very senior leadership. Plan months; meanwhile, provide DOQ-based code to users as an interim.

SOW guidance and communications

• Set expectations in SOW/RFP
  Clearly state GoldenGate is mandatory, outline approval dependencies on FTEs, include timeline buffers (weeks to months), and list prerequisites owners (billing, groups, policies, RPC/UI). Note that UI productionization is high-risk/long-lead.

• Shared documentation hub
  Slides, templates (IAM YAML, TDD), and examples will be shared in a common repo/space. Team members are encouraged to contribute lessons learned (e.g., UI constraints, Spanner access patterns).

📋Overview

• GCP project creation, billing association, and Organizer/Buganizer component setup are mandatory starters; begin pre–Day 1 to reduce delays.
• Identify GBM/FTE approvers early; many steps (IAM, GoldenGate, agent registration) require their explicit actions.
• Prepare EOG Technical Design Document with architecture diagram; prefer recommended internal tools (Spanner, SparrowQube) and seek exceptions where needed.
• UI routes beyond Gemini chat face strict RPC/stubby constraints; plan CLI/chat-first and treat rich UI as long-lead.
• Latchkey is typically enabled with billing; create IAM policy YAML and submit CLs for review and enforcement.
• GoldenGate CLI deployment to authors/Box requires FTE participation; adjust secrets/clients to first-party libs and validate RPC compatibility.
• Agent registration needs manager-filed Buganizer requests and Dolphin server configuration; approvals via LGTM/LWT.
• Final production (Elder) approval is lengthy; provide DOQ-based interim usage and reflect risks/timelines in the SOW.
• Include data security/PII controls and guardrails (pre/post-LLM) in design and SOW; document all dependencies and expirations.
