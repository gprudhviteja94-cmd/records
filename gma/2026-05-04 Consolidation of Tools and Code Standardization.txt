📒Meeting Outline

Code standardization and tool consolidation

• Current fragmentation across agents
  The team acknowledged that multiple agents have separate, duplicative tool files and helper methods (e.g., Baganizer access, test tracker, Topdate, Sponge), causing inconsistent function names, API call wrappers, and redundant code that will not pass higher-level reviews (LGT/engineering review).

• Move to a single shared tools module
  Agreement to consolidate common utilities into a central tools folder/module with a dedicated build file. Everyone should use shared methods rather than creating agent-specific variants. Even agent-unique tools should be hosted in tools to enable reuse later.

• Central documentation and ownership
  A shared document will list available tools, conventions, and usage examples. Karan to initiate the doc, covering which ports/agents use which tools, function naming patterns, and integration notes so teams can align and avoid divergence.

UAT, development status, and sequencing of changes

• Agent readiness and code review status
  CL code review completed; DT expected complete in 1–2 days; DT and TRV targeted by Wednesday; Live Devices and DL verification also cited as near-ready. Some teams remain mid-development.

• Change window alignment with UAT
  Teams with no active UAT/development can begin refactoring to shared tools immediately. Teams in active UAT may defer refactoring to avoid showstoppers, then adopt changes post-UAT.

• Staged rollout and follow-up CLs
  Current deployments can proceed as-is, followed by refactoring CLs to migrate to the common tools. Plan to start with the first four agents that have completed development and code review to validate the consolidation approach.

Governance, review, and escalation planning

• Orcas guideline and code review consistency
  Code should align with Orcas guidelines; a pass through code review is expected so shared code can merge cleanly into each agent’s framework. Past reviews were agent-specific and did not fully cover commonization; the new pass will.

• Preparation for LGT/engineering review
  Redundancy must be eliminated before LGT; engineering will scrutinize code quality, performance, and duplication. Target is to clean up within 1–2 weeks ahead of review.

• Escalation to Pallavi with plan and effort
  Before approaching Pallavi for timeline adjustments, the team will compile a concrete plan: scope, concerns, required effort, and proposed timeline extension (if any) to implement the shared tools and refactors.

Build and technical considerations

• Separate build for tools
  A separate build file for the tools module already exists. Open question on build size/line count will be revisited after analysis; growth is acceptable if it improves consistency.

• Repository structure and folder usage
  A common folder exists outside the agent section and was recently moved, but not fully adopted. Action is to start using it for all shared utilities and ensure communication so others can leverage existing code.

• Agent-to-agent collaboration
  Agents using the same tool must align on identical code paths and method names. Owners should proactively sync and avoid bespoke wrappers. Communication channels should be used to announce new/updated shared utilities.

• Continue current deployments; submit subsequent CLs for refactoring to shared tools.
