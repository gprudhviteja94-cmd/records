📒Meeting Outline

Versioning scope, goals, and phases

• Overall goal and context
  The team is introducing application versioning. Meeting focused on clarifying requirements first; detailed implementation to follow. Versioning impacts UI configuration (show/hide, labels, mandatory rules), API behavior (validations/logic), and data model.

• Phased rollout plan
  Phase 1: 
  • On homepage, fetch and display available versions; allow selection per model.
  • During “Create Model,” capture selected version; no deep UI reconfiguration in this phase.
  Phase 2:
  • Apply version-driven UI adjustments across forms/sections (e.g., RPT form), including field visibility, label changes, and mandatory/optional rules.
  • Align API validations and payloads with selected version.
  • Support duplication/migration scenarios and history.

• Clarification approach
  Requirements first; engineering to sync internally and return with implications/options. Further sessions will be scheduled for implementation challenges.

Data model for version rules: base vs delta

• Current proposal vs concern
  Current idea suggested storing a full copy of all fields per version in a collection for simplicity. Concern raised about duplication, scale (many versions), and maintenance when UI doesn’t change between versions.

• Delta-based approach (preferred direction)
  • Maintain a single immutable base configuration (v0/v1) containing all fields and defaults.
  • For each subsequent version, store only deltas (add, remove, modify, label change, mandatory toggle).
  • At runtime, resolve final config by applying version’s delta over base, then return a single consolidated JSON to the UI.
  • Benefits: minimize duplication, easier audit of changes, lower storage, clearer intent.

• Immutability and governance
  • Base config must be locked once released to avoid drift.
  • Version entries should be append-only; any correction requires a new version to protect history.
  • Consider a shared collection for UI, API, and DB-facing rules, carefully modeled to separate concerns.

UI behavior and model impacts

• UI rendering strategy
  • Ideal: backend returns final, version-resolved JSON; UI renders accordingly with minimal conditional logic.
  • Fields can be shown/hidden; labels and validations updated per version. If no delta exists for a component (e.g., RPT form), fall back to base.

• Handling removed/added fields
  • Removed fields: must not appear in the rendered form and should be excluded from the UI model submitted.
  • Added fields: appear with appropriate defaults and validations.
  • TypeScript models and form schemas must align with version selection to avoid stale/obsolete properties.

• Duplicate vs delta on the UI side
  • Team aims to avoid heavy UI duplication and prefers server-side consolidation.
  • UI maintains current behavior for Phase 1; Phase 2 adds version-aware rendering using backend-provided consolidated config.

API version handling and validations

• Version propagation
  • Selected version to be passed in the request body (not URL or headers) for create/run actions.
  • API uses version to decide validations and logic paths.

• Validation rules by version
  • Example: a field mandatory in v1 may be optional in v2; API must respect the versioned rule set.
  • If UI mistakenly sends extra fields, API will ignore fields not allowed by the selected version.

• Single endpoint, internal branching
  • Prefer single logical endpoint with internal version-aware handling over proliferating v1/v2 URLs.
  • Backward compatibility: existing runs on older versions remain valid; retention policy to govern when older versions can be deprecated.

Open questions and future considerations

• Migration and duplication scenarios
  • Behavior when duplicating a model created in v1 and opening/editing under the same or a different version.
  • Ability to switch versions mid-edit vs fixed at creation; what happens to fields that become obsolete.

• Version history and retention
  • Need a clear retention window for keeping old versions active based on existing runs.
  • Potential policy: when all runs for a version are removed, promote a newer version as the new base.

• Field identity and labeling
  • Field identities must be stable across versions while allowing label changes; deltas to capture label rename vs new field addition.

• Collections and structure
  • Likely two constructs:
    • BaseConfig (immutable after release)
    • VersionDelta (per-version, additive/removal/modification ops)
  • Ensure ops support: add, remove, modify(property set), rename(label), validation toggle.

📋Overview

• Phase 1 will surface version selection on homepage and capture version during Create Model without deep UI changes.
• Team consensus leans toward a delta-based model over full duplication to reduce redundancy and simplify maintenance.
• Backend should return a consolidated, version-resolved JSON to minimize UI conditionals and prevent stale fields.
• API will accept version in the request body and apply version-appropriate validations; single endpoint with internal branching preferred.
• Removed fields must not be included in the UI model submission; TypeScript models/schemas must adapt per version.
• Security: 3 High + 1 Critical vulnerabilities fixed; combined PR pending merge; continuous monitoring enforced.
