📒Meeting Outline

API data, JSON export, and comparison workflow

• End-to-end data flow
  The system calls an API for each RPT ID (currently 20 test cases). The API returns an “export JSON” that contains three tabs/sections (e.g., ASG, Exclusion Summary, Detail Stats/Experience Summary). For each RPT ID, the JSON is downloaded to File 1; Excel RPT (from SharePoint) is File 2. The comparison logic uses the tab “source” key in JSON to match the correct section against the Excel RPT tab.

• Source of truth and comparison targets
  The Excel RPT on SharePoint is the source of truth and should match backend data. The JSON fetched via API is compared against the Excel RPT. Output reports are generated per test case/tab (e.g., “TC1 ASG match report”) showing matches and mismatches.

• No new API required; use Python scripts
  Despite initial confusion, no new API is needed. The team will add another Python script similar to existing ones to perform comparisons. Existing scripts: “compare ASG” and “compare Exclusion Summary” already fetch JSON and compare to Excel.

New requirement: Detail Stats comparison

• Scope clarification
  A new Python script is required to compare Detail Stats JSON with the Excel RPT. This is distinct from ASG and Exclusion Summary due to year-wise data handling.

• Detail Stats multi-year handling
  Detail Stats JSON always includes five years of data (e.g., 2027–2031). The Excel must mirror this by maintaining five copies of the specific Detail Stats tab: Year 1 to Year 5. The Excel’s data is populated via a macro (“Click to Create Summary”) that must be run per year to generate each year’s table before comparison.

• Testing approach and scalability
  For development, start with 2–3 RPT IDs (e.g., T9, T11) to validate the script, then scale to all available RPT IDs. Do not hardcode a limit of 20; the solution must be scalable to any number of RPT IDs.
