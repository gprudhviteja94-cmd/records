📒 Meeting Outline

Duplicate Run Flow and User Experience

• Current duplication behavior
  The team clarified that the current system only duplicates inputs, but the intended behavior is to duplicate the full editable run details, including values changed in the previous run.
• Duplicate flow
  From the original run screen, users click Duplicate, enter a new model/run name, and then land on a Run Details screen showing the copied values from the original run.
• Editing after duplication
  Users can review and edit the duplicated values on the new screen, then either Save, Save as Draft, or Save and Run. The flow ends after this screen for the current scope.
• Draft behavior
  Auto-save creates a draft state until the user explicitly saves or saves and runs.
• Future-state items excluded for now
  The expanded top menu and keeping tabs open were identified as future enhancements and are not part of the current implementation.

Data Copying, Version Handling, and Backend Requirements

• What must be copied
  The duplicate run must include all editable fields and tabs that were changed in the original run, while keeping the original run unchanged.
• Exact value preservation
  Copied values must remain identical to the original editable values; no recalculation or transformation should occur during duplication.
• Version compatibility
  The team discussed whether a duplicated run can use the current/latest version. This depends on compatibility rules that must be defined and maintained when new versions are introduced.
• Compatibility logic
  The backend should determine which previous versions are compatible with the latest version. If a version is not compatible, the latest version option should not be shown.
• Parent-child / duplicate storage approach
  The team noted that the storage model for duplicates is still under discussion, including whether to use a parent-child structure or another approach.
• Backend and UI alignment
  The team agreed that backend and UI work should be aligned in parallel to avoid integration gaps and reduce rework.

UI Scope, Editable Tabs, and Field Behavior

• Editable tabs in duplicated run
  Only certain tabs will be enabled for editing in the duplicated run; other tabs will remain disabled/greyed out.
• Run details screen
  A new UI screen is needed to show the duplicated run details and allow editing of the copied values.
• Version dropdown
  A version selection dropdown is needed in the edit flow, and the system version is not currently present and must be added.
• Field dependencies
  Some fields follow a sequence of dependency, where changes in one field affect downstream outputs. The team discussed the need to preserve logical consistency across these dependent fields.
• FMS forecast and related fields
  There was clarification that some values, such as FMS-related inputs, are editable in specific places, while some outputs remain non-editable or are derived through calculations.

Planning, Estimation, and Next Steps

• Story breakdown
  The team agreed to break the work into smaller, technically meaningful stories rather than keeping it as one large item.
• Backend story
  A backend story will be created after the implementation approach is finalized, covering copying of editable fields and version compatibility logic.
• UI stories
  UI work will likely include the duplicate run details screen, editable tab behavior, and version dropdown support.
• Technical follow-up
  A technical discussion with the backend team will be scheduled before final estimation, so the stories can be estimated together with clearer requirements.
• Figma and reference materials
  The Figma link and feature references were shared in chat for review and implementation alignment.
