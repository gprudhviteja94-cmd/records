📒Meeting Outline

Duplication flow: goals, scope, and UI behavior

• Duplicate preserves prior run edits
  The duplicated run must retain all user edits from the original successful run. On the duplication screen, include and pre-populate the editable tabs the user had modified previously, so users can understand what they’re duplicating and re-running.
• Run details visible in duplication UI
  Include “Run Details” in the duplication screen, bringing forward the original run’s reports. In the UI, clearly highlight the tabs that were editable in the previous run (7–8 specific tabs per design). Run Details represent the original run and do not change while the duplicate is in draft.
• Version indicators and switching
  Show two version references: the version used in the original run (base) and the latest/current version. Users may switch between the base and latest versions while in draft (with auto-save). Until compatibility rules are defined, duplicates will run on the original version only. Later, if compatible, allow running on latest; otherwise, show an error.

Versioning logic and edge cases

• Limited version choices
  Only two options are presented: the base version (from which the duplicate originated) and the latest version. There is no full list of historical versions in the dropdown.
• Auto-save and UI reflection
  When a user switches the draft to the latest version, auto-save updates the model to that version. If both “version used” and “current version” become the same (e.g., V3), the dropdown still allows switching back to the other available option (base or latest) until submission.
• Run details source of truth
  Run Details always reflect the executed version and data from the original run (the base). They don’t update to the latest selection until the new run is submitted and completed.

What can and cannot be edited in a duplicate

• Formulary locking
  Do not allow changing the formulary in a duplicate. Since outputs (e.g., Rate per Quantity, RA Output) depend on formulary, changing it invalidates duplication intent. Block formulary edits to preserve forecast integrity.
• Other inputs and tabs
  • Allow changes to inputs that do not invalidate the duplicated scenario; these changes will regenerate outputs upon submission.
  • Business segment and forecast-level standard selectors likely remain fixed (non-editable); clarification to be finalized per design.
  • Weight loss, biosimilar selections (e.g., Stelara, Humira) and related flags are copied from the original run and can be adjusted in the duplicate; outputs should update accordingly after submission.
• Clarification on “outputs” vs “forecast”
  The “copied output tabs” (e.g., RA Rate per Quantity, RA Output) represent forecasted data used as editable inputs for the new run, not finalized outputs. Users may edit them; final results will be regenerated via resubmission flow.

Compatibility and calculations

• Compatibility rules (TBD)
  Compatibility criteria between versions are not yet defined. Interim rule: run duplicates on the original version only. Future logic to be introduced once versioning/compatibility rules are available.
• Engine refresh and master data
  Upcoming 22.8 refresh is expected to be master-data-only (no new calculations). Version tagging will be applied across documents. No new compatibility constraints yet.
• Resubmission behavior
  Duplicating creates a new RPT ID. After edits, submitting follows the standard resubmission flow to regenerate all tabs through the calculation engine. Until submission, discrepancies can exist between copied “forecast” tabs and changed inputs; this is acceptable and will be resolved on run.

Discoverability and identification

• UI cues to distinguish run type
  • The version dropdown appears only for duplicates (not fresh runs), serving as a UI indicator.
  • Backend maintains a “base reference RPT ID” for duplicates; null for fresh runs.
  • Enhancement request: consider a column/indicator on the RPT table to label duplicates and show base reference without opening the model (new user story to be explored with Pete).

API, persistence, and backlog management

• Save vs Submit APIs
  • Save API: persist model inputs and edited “forecast” tabs against the new RPT (auto-save and manual save).
  • Submit API: finalize and trigger the calculation engine run with the selected version and all edited data.
• Data payload requirements
  Submit flow must send both model inputs and edited tab data. Existing resubmission flow is not sufficient as-is; it needs enhancement to carry model inputs plus tab data.
• Collections and relationships
  New MongoDB collection(s) for “duplicate RPT run with full editable input preservation” to model parent-child/tab relationships. Technical details to be finalized after the scheduled Tuesday technical call.
• Story pointing and prioritization
  • Debate on API story sizing: votes clustered at 3 and 5; 8 not considered appropriate; 2 generally discouraged for API stories. Tentative majority: 5.
  • UI story to highlight previously editable tabs: discussion between 3 and 5; proponents of 5 note enable/disable logic and QA scope; lean toward majority decision after reviewing acceptance criteria.
  • Iteration hygiene: ensure stories are assigned to the current iteration (avoid “unscheduled”). Ravindra to rebalance capacity and swap lower-priority stories (e.g., aggregation) to prioritize duplication items.
