📒Meeting Outline

Dynamic UI schema and dropdown behavior

• Dropdown value issue and field types  
  ऐसा देखा गया कि एक dropdown खुला लेकिन कोई मान नहीं दिखा; अंतर्निहित फ़ील्ड में स्ट्रिंग मान था जबकि UI विकल्पों की अपेक्षा करता था। सहमति बनी कि फील्ड्स किस प्रकार के प्रकार और मान प्रदर्शित करते हैं, इसे मानकीकृत किया जाए ताकि dropdowns हमेशा सही विकल्प सूची प्राप्त करें।

• Schema-driven UI to reduce per-field mapping  
  प्रस्ताव रखा गया कि एकल, व्यक्तिगत फील्ड मैपिंग से हटकर schema-driven दृष्टिकोण अपनाया जाए। UI को backend से JSON schema पढ़ना चाहिए, जिससे लेबल, फील्ड प्रकार, विकल्प और dependencies स्वचालित रूप से प्राप्त हों। इससे UI “शुद्ध” और भविष्य के लिए सुरक्षित बनेगी: नया फील्ड जोड़ने के लिए केवल मॉडल/schema अपडेट करना होगा, UI कोड नहीं।

• Optional vs required handling  
  स्पष्ट किया गया कि schema में वैकल्पिक/आवश्यक स्थिति को एन्कोड करने की जरूरत है ताकि UI बिना हार्डकोडिंग के वैधता और दृश्यता नियम लागू कर सके। चर्चा कि क्या मौजूदा संस्करण में व्यापक बदलाव स्वीकार्य हैं।

Backend contract and JSON structure

• Backend-provided JSON schema  
  Backend एक संरचित JSON भेजेगा (सुझाव: वस्तुओं की सूची या फ़ील्ड नाम द्वारा की गई वस्तुएं) जिसमें लेबल, प्रकार, विकल्प, आवश्यक फ़्लैग, और dependency मेटाडेटा शामिल होंगे। UI केवल इस schema से फॉर्म रेंडर करेगा।

• Dependent fields representation  
  निर्णय कि dependent fields को फ़ील्ड परिभाषा के भीतर वस्तुओं की सूची के रूप में प्रस्तुत किया जाए। प्रत्येक dependency वस्तु में शर्त और child field schema शामिल होना चाहिए। केवल dependent fields स्वतंत्र रूप से नहीं दिखेंगे यदि उनके पैरेंट की शर्त पूरी नहीं होती।

• Consistency and parsing  
  ज़ोर दिया गया कि JSON अनुबंध सुसंगत और UI द्वारा आसानी से पार्स करने योग्य होना चाहिए ताकि प्रत्येक फ़ील्ड के लिए अलग-थलग हैंडलिंग न करनी पड़े। यदि सूची का उपयोग होता है, तो प्रत्येक वस्तु में पर्याप्त पहचानकर्ता होने चाहिए; यदि वस्तुओं का उपयोग होता है, तो कुंजियाँ स्थिर होनी चाहिए।

Current implementation status and scope

• Existing pages and “latest” page alignment  
  कुछ “cost” पेजों का कार्य शुरू हो चुका है। नए schema-driven दृष्टिकोण को latest पेज आर्किटेक्चर के साथ संरेखित करने की जरूरत है ताकि पुनःकार्य से बचा जा सके।

• Prototype-first approach  
  सहमति बनी कि schema-driven UI लेयर का एक त्वरित प्रोटोटाइप बनाया जाए ताकि व्यवहार्यता सत्यापित हो, डेवलपर की सुविधा की पुष्टि हो, और किनारों के मामलों (जैसे nested dependencies, array-of-objects इनपुट) का पता चले।

• Scope and change management  
  स्वीकार किया गया कि schema दृष्टिकोण एक व्यापक परिवर्तन है। टीम यह तय करेगी कि अब क्या शामिल किया जाए और क्या बाद में। निकट अवधि का लक्ष्य है कि UI केवल पढ़ने के लिए schema से लेबल, प्रकार, विकल्प, आवश्यक, और dependencies प्राप्त करे।

Data specifics and examples

• Example: ARR/Run-rate field  
  “ARR/Run-rate” फ़ील्ड समूह पर चर्चा की गई जहाँ कुछ child fields सख्ती से dependent हैं और कभी स्वतंत्र रूप से रेंडर नहीं होनी चाहिए। यह dependency एन्कोडिंग के लिए परीक्षण मामला है।

• Arrays vs objects debate  
  यह बहस हुई कि backend को फ़ील्ड वस्तुओं की सूची भेजनी चाहिए या फ़ील्ड नाम द्वारा की गई वस्तु। प्रारंभिक झुकाव सूची की ओर है ताकि क्रम सुरक्षित रहे, बशर्ते प्रत्येक फ़ील्ड वस्तु में स्थिर key हो।

• Validation and charge logic separation  
  सुझाव दिया गया कि व्यवसाय/चार्ज लॉजिक backend पर रखा जाए; UI केवल schema संकेतों के आधार पर रेंडर और वैलिडेट करे, जिससे coupling कम हो और UI स्थिति “शुद्ध” बनी रहे।

📋Overview

• Dropdown विकल्प और मान प्रकार backend schema से प्राप्त होने चाहिए ताकि खाली मान की समस्या न हो।
• प्रति-फ़ील्ड मैनुअल मैपिंग को खत्म करने और भविष्य में फील्ड जोड़ने में आसानी के लिए schema-driven UI अपनाएं।
• Backend एक सुसंगत JSON schema प्रदान करेगा जिसमें लेबल, प्रकार, विकल्प, आवश्यक, और dependencies शामिल होंगे।
• Dependent fields को dependency वस्तुओं की सूची के रूप में एन्कोड किया जाएगा; केवल dependent fields स्वतंत्र रूप से नहीं दिखेंगे।
• प्रगति में “cost” पेजों को schema दृष्टिकोण के साथ संरेखित करें और प्रोटोटाइप के माध्यम से सत्यापित करें।
• ARR/Run-rate अनुभाग को dependency हैंडलिंग और रेंडरिंग लॉजिक के लिए पायलट के रूप में उपयोग करें।
• क्रम और पार्सिंग स्पष्टता के लिए स्थिर keys वाले फ़ील्ड वस्तुओं की सूची को प्राथमिकता दें; अंतिम निर्णय प्रोटोटाइप में सत्यापित होगा।
