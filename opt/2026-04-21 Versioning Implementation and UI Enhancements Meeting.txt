📒Meeting Outline

Versioning implementation and demo status

• API-driven version creation on new RPT/template
  • New RPT/model creation now assigns a version automatically (current: 22.07 / V202.07 per database). Version value is sourced from API (Get Model by RPT ID and “current version” endpoint), no longer hardcoded in UI.
• Duplication flow with version selection
  • When duplicating an existing RPT, a version dropdown is displayed. For demo, two versions appear but only current version is active. Selected version is sent on autosave/save-as-template/run, to drive UI refresh aligned to version-specific field visibility.
• Static fields by version via API integration
  • Static fields are now returned per version from API. Future additions/removals of fields will be governed by selected version, enabling UI to reflect version-based schema changes.
• Data source and consistency
  • Demo moved from dummy/hardcoded to live database values. Newly created cases/models will carry version V202.07, consistent with the team’s Excel source of truth.

Current limitations and plan

• Partial completion of versioning stories
  • Most versioning stories are in progress; full end-to-end version switching UX (including dynamic field add/remove and visibility refresh) not yet demoable due to having only one effective version.
• Version history page and QA status
  • QA and UI teams report no current issues on the version history page. Further comprehensive demo planned once the full “watch-to-end” flow is complete.
• Next sprint target
  • Remaining versioning stories expected to be completed and re-demoed next sprint.

List view enhancements

• Sortable columns for RPT list
  • Users can now sort by: RPT Model Name, Run Date, RPT ID, Basic Definition, Specialty Definition. Sorting toggles ascending/descending (e.g., A–Z then Z–A), applied consistently across supported columns.
